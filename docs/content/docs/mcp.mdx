The Memsy [MCP](https://modelcontextprotocol.io) server turns any MCP-compatible host into a Memsy-aware agent. One server, every host.

## What you get

8 tools your AI agent can call:

| Tool | Purpose |
|---|---|
| `memsy_search` | Semantic search across stored memories. |
| `memsy_ingest` | Store events (chat turns, decisions, facts) — batched up to 100. |
| `memsy_status` | Confirm async ingest finished extraction. |
| `memsy_health` | Connectivity + version check. |
| `memsy_list_memories` | Paginated browse over the console memory store. |
| `memsy_get_memory` | Fetch one memory by ID. |
| `memsy_list_orgs` | List configured Memsy profiles (one profile = one org). |
| `memsy_use_org` | Switch which Memsy org subsequent calls hit. |

Plus 4 resources — `memsy://memories/recent`, `actor/current`, `session/current`, `profile/current` — and 2 prompts — `recall-context`, `summarize-and-store`.

<Info>
Requires Node.js 18+. You don't need to install anything globally — your host spawns the server via `npx`.
</Info>

## Quickstart

<Steps>
  <Step title="Get your API key">
    Open **[app.memsy.io](https://app.memsy.io)** → Settings → API Keys → **Create key**. It looks like `msy_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx`.

    See [Sign Up](/docs/signup) if this is your first time.
  </Step>

  <Step title="Add the MCP server to your host">
    Paste this into your host's MCP config, replacing the placeholder key.

    <Tabs>
      <Tab title="Claude Code">
        Add to `~/.claude.json` (user-level) or `.claude/settings.json` (per-project):

        ```json title="~/.claude.json"
        {
          "mcpServers": {
            "memsy": {
              "command": "npx",
              "args": ["-y", "@memsy-io/mcp"],
              "env": {
                "MEMSY_API_KEY": "msy_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
              }
            }
          }
        }
        ```
      </Tab>

      <Tab title="Cursor">
        Add to `~/.cursor/mcp.json` (global) or `.cursor/mcp.json` (per-project):

        ```json title=".cursor/mcp.json"
        {
          "mcpServers": {
            "memsy": {
              "command": "npx",
              "args": ["-y", "@memsy-io/mcp"],
              "env": {
                "MEMSY_API_KEY": "msy_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
              }
            }
          }
        }
        ```
      </Tab>

      <Tab title="Cline">
        Open Cline → Settings → MCP Servers → **Add**. Paste:

        ```json
        {
          "memsy": {
            "command": "npx",
            "args": ["-y", "@memsy-io/mcp"],
            "env": {
              "MEMSY_API_KEY": "msy_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
            }
          }
        }
        ```
      </Tab>

      <Tab title="Continue.dev">
        Add to `~/.continue/config.yaml`:

        ```yaml title="~/.continue/config.yaml"
        mcpServers:
          memsy:
            command: npx
            args: ["-y", "@memsy-io/mcp"]
            env:
              MEMSY_API_KEY: msy_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
        ```
      </Tab>

      <Tab title="Zed">
        Add to `~/.config/zed/settings.json` under `context_servers`:

        ```json title="~/.config/zed/settings.json"
        {
          "context_servers": {
            "memsy": {
              "command": {
                "path": "npx",
                "args": ["-y", "@memsy-io/mcp"]
              },
              "env": {
                "MEMSY_API_KEY": "msy_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
              }
            }
          }
        }
        ```
      </Tab>
    </Tabs>
  </Step>

  <Step title="Restart your host">
    MCP config is read on launch.
  </Step>

  <Step title="Verify it's connected">
    <Tabs>
      <Tab title="Claude Code">
        Type `/mcp` — `memsy` should appear with 8 tools.
      </Tab>
      <Tab title="Cursor">
        Settings → Tools & MCP → `memsy` should show a green status dot.
      </Tab>
      <Tab title="Cline">
        Bottom panel → MCP icon → `memsy` listed.
      </Tab>
      <Tab title="Continue.dev">
        Type `@` in chat — Memsy tools should be discoverable.
      </Tab>
    </Tabs>

    Then ask your agent *"use memsy_health to check the connection"*. Expected:

    ```json
    { "status": "ok", "version": "...", "base_url": "https://api.memsy.io/v1" }
    ```

    You're done.
  </Step>
</Steps>

## Using it day-to-day

Talk to your agent naturally — it picks which tool to call.

**Recall**:
- *"What did we decide about billing storage?"*
- *"Search memsy for context on the auth migration."*
- *"Did we discuss why we use Postgres over Mongo?"*

**Store**:
- *"Remember that we use bcrypt for password hashing, not argon2."*
- *"Note: the SSO migration is blocked on the Clerk webhook fix."*
- *"Save this decision: we're shipping the freemium tier before the team plan."*

**Browse / manage**:
- *"List my recent memories."*
- *"Show me memory mem_abc123."*

The agent translates these into the right `memsy_search` / `memsy_ingest` / `memsy_list_memories` calls.

## Multi-org setup

If you work across multiple Memsy orgs (personal + work, multiple clients), use a config file with named profiles instead of putting the key in `env`.

<Steps>
  <Step title="Create ~/.memsy/config.json">
    ```json title="~/.memsy/config.json"
    {
      "active_profile": "personal",
      "profiles": {
        "personal": {
          "api_key": "msy_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
          "base_url": "https://api.memsy.io/v1",
          "org_label": "Personal"
        },
        "work": {
          "api_key": "msy_yyyyyyyyyyyyyyyyyyyyyyyyyyyyyyyy",
          "base_url": "https://api.memsy.io/v1",
          "org_label": "Calfus Engineering"
        }
      }
    }
    ```
  </Step>

  <Step title="Lock down permissions">
    ```bash
    chmod 600 ~/.memsy/config.json
    ```

    The server prints a stderr warning if it's world-readable.
  </Step>

  <Step title="Drop the API key from your host's MCP env">
    ```json
    {
      "mcpServers": {
        "memsy": {
          "command": "npx",
          "args": ["-y", "@memsy-io/mcp"]
        }
      }
    }
    ```

    The server discovers profiles from the config file.
  </Step>

  <Step title="Switch orgs at runtime">
    Ask the agent: *"switch memsy to the work profile"* — it calls `memsy_use_org { profile: "work" }`. Verify with `memsy://profile/current` or *"what's the active memsy profile?"*.

    A per-project `<project>/.memsy/config.json` overrides the per-user one — useful for repo-specific defaults.
  </Step>
</Steps>

## Configuration reference

### Environment variables

| Variable | Default | Purpose |
|---|---|---|
| `MEMSY_API_KEY` | — | API key. Synthesizes a `default` profile when no config file exists. |
| `MEMSY_BASE_URL` | `https://api.memsy.io/v1` | Memsy hot-path API. Override for staging or self-hosted. |
| `MEMSY_PROFILE` | `default` | Which named profile to activate at startup. |
| `MEMSY_ACTOR_ID` | derived | Override the actor identity. |
| `MEMSY_DEFAULT_ROLE_IDS` | — | Comma-separated role IDs applied as default search filters. |
| `MEMSY_DEFAULT_TEAM_IDS` | — | Comma-separated team IDs applied as default search filters. |

### CLI flags

```bash
npx @memsy-io/mcp --profile work
npx @memsy-io/mcp --api-key msy_... --base-url https://staging.memsy.io/v1
npx @memsy-io/mcp --config /path/to/config.json
```

CLI flags beat env vars beat config-file `active_profile`.

### Resolution order

- **API key**: `--api-key` → `MEMSY_API_KEY` → active profile's `api_key`
- **Profile name**: `--profile` → `MEMSY_PROFILE` → config's `active_profile` → `default`
- **Config file path**: `--config` → `./.memsy/config.json` → `~/.memsy/config.json`

## Identity model

`actor_id` scopes who memories belong to. The server derives a stable, deterministic value:

```
actor_id = sha256("<profile_name>|<git config user.email>")[:16]
```

No PII to the server — just the hash. Same developer, same email, same profile, same repo → same actor_id across sessions.

**Defaults**:

- **Ingest** tags new events with this derived `actor_id` so you can tell which were stored via this MCP.
- **Search** is **org-wide** by default — finds memories regardless of which channel stored them (dashboard, SDK, prior sessions). Pass `actor_id` explicitly to scope down.

Override with `MEMSY_ACTOR_ID=neel-shah` (env) or per-profile `actor_id` in config. See [Actors and Sessions](/docs/actors-and-sessions) for the broader concept.

## Troubleshooting

### `MemsyAuthError` or 401

API key missing, malformed, or revoked. Check:

- `MEMSY_API_KEY` is in the host's MCP `env` block (not just your shell — the host doesn't inherit shell env).
- Key starts with `msy_` and has 32 hex chars.
- Key is active in app.memsy.io → API Keys.

### `MemsyAPIError` 404

Wrong `base_url`. Production hot-path is `https://api.memsy.io/v1` — the `/v1` suffix is required. The server defaults to it; override only for staging or self-hosted.

### `memsy_search` returns `count: 0` but memories exist

The query may not semantically match. Try:

- Broader terms / single keyword.
- `memsy_list_memories` (no args) to see what's actually stored.
- `threshold: 0` (the default — anything stricter hides weak matches).

Search is org-wide by default, so prior-channel memories are findable without setting `actor_id`.

### Server doesn't start / host shows "memsy: disconnected"

Run the server manually to see the error:

```bash
MEMSY_API_KEY=msy_... npx -y @memsy-io/mcp
```

Common causes:

- No key + no config file → "No Memsy profile resolved"
- Config file is invalid JSON → parse error on stderr
- Node 17 or older → upgrade to 18+

### Profile switch doesn't persist across sessions

`memsy_use_org` swaps the active profile in the current MCP process only. Each host spawns a fresh process per chat. To change the default at startup, set `MEMSY_PROFILE` in the host's MCP `env`, or update `active_profile` in the config file.

## Security notes

- The config file stores API keys in plain text. `chmod 600` is enforced via stderr warning. Use a passworded keychain manager if your threat model demands it.
- API keys are held in process memory and sent only to your configured `base_url` over HTTPS. They are never logged.
- `memsy_list_orgs` and `memsy://profile/current` never return the API key — only metadata.
